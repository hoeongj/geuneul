# ADR-0003. 좌표 미제공 공공데이터는 geocode-before-insert + 저장 좌표 재사용

- 상태: 승인 (2026-07-02)
- 관련: `KakaoGeocodingClient`, `IngestionService`(4단계 파이프라인), `places.geocoded`

## 문제(Context)

2025-02 이후 전국공중화장실표준데이터가 **WGS84 좌표를 제공하지 않는다** — 실측 파일(59,768행, 34컬럼)에
위도/경도 컬럼이 아예 없음을 확인했다. 주소(도로명/지번)는 100% 존재한다.
좌표 없는 행은 공간검색(그늘의 간판)에 올릴 수 없으므로 **주소→좌표 지오코딩이 필수 경로**가 됐다.

제약: 카카오 로컬 API는 무료 쿼터·QPS 제한이 있고, 6만 건 외부 HTTP 호출은 수십 분 걸린다.

## 결정(Decision)

1. **geocode-before-insert** — geom을 nullable로 풀지 않는다. 좌표를 얻은 행만 삽입해
   "places에 있으면 반드시 지도에 찍힌다"는 공간 불변식을 유지. 실패 행은 계수·로그로 남긴다.
2. **저장 좌표 재사용** — 지오코딩 산출 행은 `geocoded=true`로 저장하고, 재적재 시
   `(source, external_id)`가 존재하고 **주소가 안 바뀌었으면 재호출 없이 스킵**.
   → 주기 동기화(P3)가 카카오 쿼터를 다시 태우지 않는다. IT로 검증(2차 실행 시 성공분 호출 0회).
3. **트랜잭션 경계 = upsert 단위** (ADR-0002 개정) — 60k 외부 호출을 단일 트랜잭션에 넣으면
   커넥션 점유·롤백 폭이 과대. 파이프라인을 ①파싱 ②좌표보유분 upsert(tx) ③지오코딩(무tx)
   ④산출분 upsert(tx)로 분리. 부분 실패해도 멱등 upsert라 재실행으로 수렴.
4. **동시성 = 가상 스레드 + Semaphore(8)** — I/O 대기는 가상 스레드로 넓게, 카카오 QPS는
   세마포어 상한으로 보호. 429는 백오프(500ms×시도차수) 후 **총 3회까지 시도**(최대 재시도 2회).
5. 도로명(road_address) 매칭 우선 — 표준데이터 주소가 도로명 기준이라 정확도가 높다.

## 검토한 대안(Alternatives)

| 대안 | 기각 이유 |
|---|---|
| geom nullable + 나중에 채우기 | 공간 불변식 파괴 — 검색에 안 나오는 유령 행. NULL 처리 분기가 전 쿼리로 전파 |
| 지오코딩 결과 별도 캐시 테이블 | places 자체가 (자연키→좌표) 저장소 역할을 이미 함. 테이블 중복 |
| VWorld/국토부 지오코더 | 카카오가 이미 지도 SDK로 채택돼 있어 정렬. 나머지는 백업(쿼터 소진 시) |
| 배치 프레임워크(Spring Batch) 도입 | 단일 소스·단발 실행에 과설계. 스케줄·재시도 요구가 커지는 P3에서 재검토 |

## 결과(Consequences)

- 좌표 미제공 포맷 60k행 중 지오코딩 성공분(프로덕션 실측 46,897행)이 공간검색 가능한 데이터로 변환된다 (실행은 KAKAO_REST_API_KEY 필요; 무효 주소 등 실패분은 재시도 대상).
- 주소가 바뀐 행만 재지오코딩 — 동기화 비용이 변경량에 비례.
- 한계: 지오코딩 실패 행(무효 주소)은 매 동기화마다 재시도된다(성공 캐시만 있음). 실패 캐시는
  실패 사유 구분(일시적/영구적)이 필요해지면 추가.
